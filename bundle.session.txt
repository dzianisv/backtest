  The deployer story is genuinely good. The builder loop is fine. The
  interactive first-touch is missing entirely — and that's the one your question
  is about.

  My opinion: the primary verb is ask, not run a workflow

  Workflows are an implementation detail. A user should never have to know
  hedge-fund-committee.workflow.js exists. The right mental model:

  User: "should I buy BTC today?"
    → Claude Code reads AGENTS.md, routes to crypto-chair
    → chair fans out to research-desk/onchain/liquidity skills
    → (heavy question) fires the panel workflow underneath
    → returns research.crypto.{date}.md

  The workflow fires underneath a question. It is not the front door. Forcing
  users to "start a workflow" is like making them invoke a function instead of
  asking a question — wrong altitude for a recommend-only advisor.

  Why not the wizard (Option B)

  A /start wizard is ceremony that earns its keep only for the deployer
  (scheduling, notify wiring) — and setup-*.md already does that job. For the
  interactive user it's friction in front of a thing that already works via
  plain language.

  What I'd actually build

  One QUICKSTART.md + one routing contract in AGENTS.md. No new runtime code.

  1. QUICKSTART.md — the 60-second "I cloned this, now what":
    - Just ask (3 example questions per workstream)
    - Want it daily/proactive? → point to setup-*.md
    - Want to improve the engine? → point to the loop
  2. A routing table in AGENTS.md that maps question-shapes → skill/workflow, so
  Claude reliably dispatches "should I buy X" to the chair instead of
  freelancing. This is the actual load-bearing piece — without it, the
  natural-language UX is luck, not design.

  The single most important sentence the repo is missing: "Open this folder in 
  Claude Code and ask your question — examples below." Everything else is depth
  behind that door.

  Want me to write QUICKSTART.md + the AGENTS.md routing table? That's ~30 min
  and closes the real gap. I'd skip the wizard.
